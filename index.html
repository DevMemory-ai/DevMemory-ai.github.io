How DevMemory tracks developer intent, audits your builds, and injects context into your IDE.

Step 1 — The Watcher Daemon

The core daemon runs continuously in the background of your workspace. With one command, it starts watching your files and directory structure, detecting edits, file saves, and staged changes.

TERMINAL 
devmemory watch

Step 2 — Commit detection

Every time you commit changes via git or trigger a build, DevMemory intercepts the action, captures the git diff, parses the changed lines, and retrieves your development logs.

Step 3 — The 7-agent pipeline

Your diffs are analyzed by seven specialized local or cloud agents. Here is what they build:

  • Summarizer — Reads the diff and explains what changed and why in plain English.
  • Architect — Generates a Mermaid.js data flow diagram showing affected modules.
  • Security — Scans for hardcoded credentials, input validation flaws, and logic exploits.
  • Tester — Builds a verification plan with mock inputs and validation scripts.
  • Regenerator — Writes the exact prompt needed to rebuild this feature from scratch in any AI tool.
  • Changelog — Auto-updates your project changelog with human-readable entries.
  • Brain — Updates the living project-brain.md that your AI reads at every session start.

Step 4 — The .devmemory/ vault

All structured outputs are stored in a local `.devmemory/` vault on your computer. This vault serves as a secure knowledge base containing your historical design decisions.

DIRECTORY STRUCTURE 
.devmemory/
├── config.json
├── brain/
│   └── project-brain.md
├── builds/
│   ├── 2026-06-16_01_user-auth.md
│   └── 2026-06-16_02_payment-gateway.md
└── logs/
    └── daemon.log

Step 5 — MCP context injection (Pro)

The Pro MCP Server interfaces directly with your IDE (Cursor, Claude Code, VS Code, Windsurf). When you boot up a new session, the IDE queries the server, loading the `.devmemory/` vault contents into the active LLM context so the assistant knows exactly where you left off.

Frequently Asked Questions

Q: Does DevMemory send my code to the cloud?

A: No. The free CLI stores everything locally in .devmemory/ on your machine. The Pro MCP server runs locally too — nothing leaves your computer unless you explicitly enable cloud sync.

Q: Which AI tools does it work with?

A: Cursor, Claude Code, Windsurf, VS Code + Cline, and Aider. Any tool that supports MCP works with the Pro server. The free CLI works with any AI tool via copy-paste.

Q: Can I use it offline?

A: Yes. DevMemory supports Ollama for 100% offline, credit-free operation. No internet required.

Q: What's the difference between the CLI and the MCP server?

A: The free CLI captures and documents your builds locally. The Pro MCP server goes further — it connects your .devmemory/ vault directly to your IDE so your AI reads it automatically at every session start. No copy-pasting required.

Q: Is DevMemory open source?

A: The core CLI is fully open source under MIT license. The Pro MCP server is commercial. You can inspect, fork, and contribute to the CLI on GitHub.

A1. Introduction

Modern AI coding tools like Cursor, Claude Code, and Aider write features at blazing speed. However, they lack long-term memory. After a few days, developers forget why architectural decisions were made, how a feature runs, and what original prompts generated it.

DevMemory AI is an agentic developer tool that automatically captures and documents every AI-generated build. For each completed feature or commit, DevMemory AI creates structured memory files containing:

  • The original developer prompt (the intent behind the change).
  • A summary of exactly what was modified, added, or removed.
  • A Mermaid.js architecture diagram showing data flow and component interactions.
  • A security audit checking for hardcoded keys, logic flaws, and data sanitization risks.
  • A validation plan with test cases, verification steps, and mock inputs.
  • A regeneration prompt you can copy-paste into any AI to recreate the feature from scratch.
THE MEMORY MANIFESTO

Git tracks code. DevMemory tracks intent. Git stores what changed. DevMemory stores why it changed, how it works, and how to reproduce it.

The goal is to make every AI-generated feature understandable, reproducible, searchable, and reusable — even months after it was written.

A2. Prerequisites

Before installing DevMemory AI, you need two tools installed on your machine and available in your system PATH. Without these, the CLI will not function.

Node.js 18+

Version 18.0.0 or higher is required. DevMemory is built on Node.js and distributed via npm.

↗ Download Node.js

Git

Git must be installed and initialized in your project. DevMemory reads git diffs, commit history, and staging areas to analyze your code changes.

↗ Download Git

Verifying Your Installation

Open a terminal and run these commands to confirm both tools are accessible:

TERMINAL 
node --version
# Expected output: v18.0.0 or higher

git --version
# Expected output: git version 2.x.x

Node.js includes npm by default. You can verify npm is available by running npm --version.

A3. Installation

DevMemory AI is distributed as a global npm package. Installing it globally registers the devmemory command system-wide.

⚠ IMPORTANT PRIVILEGES NOTE
Because DevMemory AI registers global binaries and configures background watcher daemons (launchd plist, systemd user services, or startup batch files), you must run the installation command with Administrator/Superuser privileges.

macOS & Linux (using sudo):

TERMINAL 
sudo npm install -g devmemory-ai

Windows (Administrator mode): Open PowerShell or Command Prompt as Administrator and run:

POWERSHELL (ADMIN) 
npm install -g devmemory-ai

What Happens During Installation

  1. npm downloads the devmemory-ai package and registers the global devmemory binary.
  2. The binary is ready to use immediately — no additional setup required.
  3. All dependencies are pre-built and included in the package.

Verify Installation

After installing, confirm the CLI is working by running:

TERMINAL 
devmemory --help

You should see the help menu listing all available subcommands: init, config, record, watch, show, context, setup-ollama, and terminate.

A5. devmemory init

Initializes DevMemory AI in your project repository. This is the first command you must run before using any other DevMemory features. It sets up the configuration, creates the memory vault directory, and prepares your project for autonomous documentation.

Syntax

TERMINAL 
devmemory init

No flags or arguments. Must be run from the root directory of your git repository.

What Happens When You Run It

  1. Interactive Configuration Wizard launches — You are prompted to select your AI provider, enter your API key, choose a model, and set a cooldown timer.
  2. Creates the .devmemory/ directory — All build logs, brain, changelog, and context live here.
  3. Saves .devmemory/config.json — Local provider and key configurations.
  4. Generates .devmemory/project-brain.md — codebase template.
  5. Generates .devmemory/changelog.md — project changelog template.
  6. Updates .gitignore — hides API key storage.
  7. Registers background daemon — launchd (macOS), systemd (Linux), Startup folder (Windows).

Example Session

TERMINAL — EXAMPLE OUTPUT 
$ cd ~/my-project
$ devmemory init
Initializing DevMemory AI...
=== DevMemory AI Configuration Setup ===

Available API Providers:
  1. Google Gemini
  2. OpenAI
  ...
Choose a provider index [1]: 1
Enter API Key for Google Gemini: ********
Configuration completed successfully!
Created initial .devmemory/project-brain.md
[DevMemory] Background daemon registered successfully! (Label: com.devmemory.watcher.b36611f299d9)

A6. devmemory config

Re-runs the interactive configuration wizard. Use this command when you need to change your AI provider, swap API keys, switch models, or adjust the watcher cooldown time — without re-initializing the entire project.

Syntax

TERMINAL

No flags or arguments. Must be run from the root directory of your project.

A7. devmemory record

Manually triggers a one-time build memory capture from your current uncommitted changes. This is the on-demand alternative to the continuous watch daemon.

Syntax

TERMINAL 
# Basic usage (will prompt for description interactively)
devmemory record

# With an inline prompt flag
devmemory record --prompt "Add JWT user login validation"

# Short flag version
devmemory record -p "Fix database connection pooling"

Flags

FlagShortDescription
--prompt-pThe developer intent description. If omitted, prompts interactively.

A8. devmemory watch

Starts the continuous background watcher daemon. This is the core runtime mode of DevMemory AI.

Syntax

TERMINAL 
devmemory watch

How It Works

Monitors the filesystem. When you save edits or commit changes, it runs the multi-agent AI pipeline and saves build metadata, updates the codebase brain, and updates your changelogs.

Failure Resilience & Alerts

If the LLM connection fails, DevMemory will pop up a native OS alert to notify you (via AppleScript on macOS, PowerShell on Windows, zenity on Linux) and back off exponentially to protect CPU and budgets.

A9. devmemory show

Renders and prints a markdown memory file beautifully in the terminal using ANSI color formatting.

Syntax

TERMINAL 
devmemory show <target>

Arguments

  • <target>: Can be a build number (e.g. show 3), brain, changelog, context, or a markdown file path.

A15. Rate Limiting

Safeguards your API billing and hardware by enforcing maximum call thresholds per session, showing native alerts, and shutting down if limits are breached.

A16. Multi-Agent Pipeline

Every build runs 7 specialized AI agents:

  • Summary Agent: Summarizes git diff changes.
  • Architecture Agent: Renders a Mermaid diagram of dependencies.
  • Regeneration Agent: Compiles prompt to rebuild from scratch.
  • Validation Agent: Designs testing cases.
  • Security Agent: Performs local code vulnerability auditing.
  • Brain Agent: Updates the global project-brain.md file.
  • Changelog Agent: Appends entries in changelog.md.

B3. IDE Configuration (Coming Soon)

IDE configuration is completely automated using prompt-based setup. Rather than editing configuration files manually, you can generate and run a simple setup prompt in your AI assistant (Cursor, Cline, Claude Code, etc.). The assistant will call the appropriate tools to configure the MCP settings globally.

⚡ AUTOMATED PROMPT SETUP

Use our interactive Prompt Generator tool to customize your setup or edit commands, then copy and paste the prompt directly into your AI chat window. The assistant will handle the rest!

1. Global Setup Prompt

Copy-paste this prompt into your chat window, replacing the bracketed values with your own details, to configure DevMemory globally:

SETUP PROMPT 
Help me configure DevMemory globally in my system. Please invoke the `devmemory_init` tool with `global: true` using the following details:

- Provider: [e.g. gemini, openai, anthropic]
- API Key: [your_api_key_here]
- Model: [e.g. gemini-2.5-flash, gpt-4o, claude-3-5-sonnet]
- Rate Limit: [Optional: e.g. 0. Keep 0 or leave empty if you want unlimited rate limits]
- Cooldown: [Optional: e.g. 10. Idle cooldown in seconds before recording. Defaults to 10]

Once initialized globally, DevMemory will automatically configure all of my current and future projects, copying this configuration locally and spinning up the background watcher daemon automatically whenever I open a new project.

2. Edit Configuration Prompt

Copy-paste this prompt to edit or update an existing configuration (API Key, Model, Rate Limits, or Cooldown) globally or locally:

EDIT CONFIG PROMPT 
Help me edit my DevMemory configuration. Please call the `devmemory_init` tool to update the configuration with the following details:

- Provider: [Optional: provider name]
- API Key: [Optional: new_api_key]
- Model: [Optional: new_model_name]
- Rate Limit: [Optional: new_rate_limit or 0 for unlimited]
- Cooldown: [Optional: new_cooldown_seconds]
- Global Update: [Optional: true to update ~/.devmemory_config.json, false to update local .devmemory/config.json]

Update only the requested options and leave the remaining configuration options unchanged.

What Happens Under the Hood?

  1. The AI assistant calls the devmemory_init tool with the merged parameters.
  2. The MCP server writes the configuration globally (~/.devmemory_config.json) or locally (.devmemory/config.json).
  3. If configured globally, the configuration is automatically synchronized locally whenever a new project workspace is opened in your IDE.
  4. The MCP server automatically spins up the background watcher daemon (devmemory watch) for the project folder so that you don't have to manually start it.
