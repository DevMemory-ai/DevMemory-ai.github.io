How DevMemory Works

An automated background memory layer that operates entirely locally.

The core watcher runs locally in the background of your workspace. With one CLI command, it starts watching your files and directory structure, automatically detecting saved changes, staged edits, and deleted files completely offline.

Frequently Asked Questions

Q: Does DevMemory send my code to the cloud?

A: No. The free CLI stores everything locally in .devmemory/ on your machine. The Pro MCP server runs locally too — nothing leaves your computer unless you explicitly enable cloud sync.

Q: Which AI tools does it work with?

A: Cursor, Claude Code, Windsurf, VS Code + Cline, and Aider. Any tool that supports MCP works with the Pro server. The free CLI works with any AI tool via copy-paste.

Q: Can I use it offline?

A: Yes. DevMemory supports Ollama for 100% offline, credit-free operation. No internet required.

Q: What's the difference between the CLI and the MCP server?

A: The free CLI is a local-only tool built from the open-source codebase on GitHub (no dashboard interface, no direct MCP integration). You can use it to capture and document builds via terminal. The Pro MCP server integrates directly with your IDE (Cursor, Claude Code, Cline, Windsurf, etc.) to automatically read/write memories, and includes our premium local Web Dashboard & Analytics UI.

Q: Is DevMemory open source?

A: The core CLI is fully open source under GNU General Public License v3.0 (GPL-3.0). The Pro MCP server is commercial. You can inspect, fork, and contribute to the CLI on GitHub.

A1. Introduction

Modern AI coding tools like Cursor, Claude Code, and Aider write features at blazing speed. However, they lack long-term memory. After a few days, developers forget why architectural decisions were made, how a feature runs, and what original prompts generated it.

DevMemory AI is an agentic developer tool that automatically captures and documents every AI-generated build. For each completed feature or commit, DevMemory AI creates structured memory files containing:

  • The original developer prompt (the intent behind the change).
  • A summary of exactly what was modified, added, or removed.
  • A Mermaid.js architecture diagram showing data flow and component interactions.
  • A security audit checking for hardcoded keys, logic flaws, and data sanitization risks.
  • A validation plan with test cases, verification steps, and mock inputs.
  • A regeneration prompt you can copy-paste into any AI to recreate the feature from scratch.
THE MEMORY MANIFESTO

Git tracks code. DevMemory tracks intent. Git stores what changed. DevMemory stores why it changed, how it works, and how to reproduce it.

The goal is to make every AI-generated feature understandable, reproducible, searchable, and reusable — even months after it was written.

Open Source GPL-3.0 Engine

The core CLI documentation engine is fully open-source and released under the GNU General Public License v3.0 (GPL-3.0). This means it is free to inspect, run locally, modify, and redistribute. It requires no license keys or online authentication, offering lifetime utility for developers who prefer an entirely offline, open-source documentation workflow.

A5. devmemory init

Initializes DevMemory AI in your project repository. This is the first command you must run before using any other DevMemory features. It sets up the configuration, creates the memory vault directory, and prepares your project for autonomous documentation.

Syntax

TERMINAL 
devmemory init

No flags or arguments. Must be run from the root directory of your git repository.

What Happens When You Run It

  1. Interactive Configuration Wizard launches — You are prompted to select your AI provider, enter your API key, choose a model, and set a cooldown timer.
  2. Creates the .devmemory/ directory — All build logs, brain, changelog, and context live here.
  3. Saves .devmemory/config.json — Local provider and key configurations.
  4. Generates .devmemory/project-brain.md — codebase template.
  5. Generates .devmemory/changelog.md — project changelog template.
  6. Updates .gitignore — hides API key storage.
  7. Registers background daemon — launchd (macOS), systemd (Linux), Startup folder (Windows).

B1. Pro Overview

DevMemory Pro connects your project's memory vault directly to your favorite AI assistant in your IDE using the Model Context Protocol (MCP).

With Pro, there is no need to manually copy-paste the FULL_PROJECT_CONTEXT.md file anymore. When you open a project in any MCP-compatible IDE or AI client (including Cursor, Claude Code, Cline, Windsurf, Claude Desktop, Aider, Antigravity, and any other tool that supports MCP), the AI assistant will automatically read your project memory, update the brain, and record builds in real-time.

B2. License Activation

DevMemory Pro utilizes a secure Dual-Verification Licensing Model to ensure authentic distribution and secure device configuration:

The Dual-Verification Process

  • Step 1: Download Gating (Web Verification): Access to the installer packages and portable zip binaries on the website is guarded by active license key validation. You must input your valid license key to download the desktop application files.
  • Step 2: Native App Activation (Device Binding): After installing and launching the desktop app, you will register your license key again. This registers your device fingerprint in the database, locks in your seat configuration, and initializes the local MCP daemon on port 31414.

Desktop Application Activation Steps

Follow these steps to download and activate DevMemory Pro:

  1. Navigate to the section of this website.
  2. Click the download button for your platform, enter your license key in the verification popup, and complete the download.
  3. Install and launch the desktop application. The License Activation screen will appear automatically on first run.
  4. Paste your license key in the input field and click Activate License to register the hardware fingerprint.
  5. DevMemory Cloud will validate the key, issue a secure JWT local token, unlock the local dashboard, and start the local MCP server on port 31414.

How Pro Licensing Works

DevMemory Pro utilizes a local-first, privacy-respecting licensing system:

  • Local MCP Port: Port 31414 is exclusively reserved for the local MCP server daemon. Once activated, IDEs connect to this local port directly.
  • Device Seats Limit: Each lifetime license key supports activation on up to 2 active devices (e.g., your personal desktop and work laptop).
  • Hardware Binding: During activation, the application queries a unique, anonymized hardware fingerprint based on your system motherboard serial number, CPU ID, or BIOS ID to associate the key to that machine without collecting personal identifying data.
  • Cryptographic Local Token: Upon successful database verification, our cloud server signs an activation token (JWT token) and stores it securely inside ~/.devmemory_secure_token.json using AES-256-GCM encryption with a key derived from your hardware fingerprint.
  • Offline Grace Period: Once activated, DevMemory operates entirely local and offline. The local server validates the cryptographic signature of the local JWT token offline. It only needs an online ping once every 7 days to refresh the grace period, meaning you can code in airplanes or remote cabins without connectivity.
  • System Clock Rollback Protection: To prevent rolling back your operating system time to bypass grace period limits, the background watcher logs timestamp checkpoints. Clock tampering automatically locks the dashboard until an online refresh is completed.

What Happens Under the Hood?

  1. The AI assistant calls the devmemory_init tool with the merged parameters.
  2. The MCP server writes the configuration globally (~/.devmemory_config.json) or locally (.devmemory/config.json).
  3. If configured globally, the configuration is automatically synchronized locally whenever a new project workspace is opened in your IDE.
  4. The MCP server automatically spins up the background watcher daemon (devmemory watch) for the project folder so that you don't have to manually start it.

B4. Auto-Initialization

Once configured, the MCP server performs silent auto-initialization. Whenever you open any folder in your IDE:

  • The MCP server checks if the local .devmemory/ repository directory exists.
  • If missing, it creates the folder structures, initialized project-brain.md, changelog.md, and timeline.json documents automatically.
  • You never have to run devmemory init manually in a terminal again — just open a workspace and code.

B5. Global Config

Running devmemory login writes your API configuration preferences globally to ~/.devmemory_config.json.

Every new workspace created or opened in your IDE automatically inherits your global keys and settings. You can still customize a project's settings by adding a local .devmemory/config.json which takes priority.

B9. Local Dashboard & Analytics

DevMemory includes a premium, glassmorphism-styled Web Dashboard & Analytics interface that gives you local visual access to all your project memories. The dashboard is fully interactive, featuring timeline charts, commit records, dependency graphs, security audits, and configuration settings editors.

Access stats, timelines, build reports, and markdown files locally in your browser. All data remains 100% offline and stays on your machine.

Accessing the Dashboard

The dashboard is served natively inside the DevMemory Desktop Application. When launched, it opens directly as a visual desktop workspace. Advanced users can also open the dashboard in any web browser at: 

http://localhost:31415

The dashboard automatically discovers all DevMemory-initialized projects on your machine. Use the project selector dropdown to switch between projects.

License Verification Guard: If no valid license key has been activated, launching the desktop application (or navigating to port 31415) will automatically render the License Activation interface to allow registration.

Running the Dashboard

The dashboard server runs locally on a dedicated port (default 31415) managed by the Desktop Application. If the port is already in use, it will automatically scan surrounding ports (up to 31465) to bind successfully. You can also launch the CLI dashboard server manually using: 

devmemory dashboard

Or specify a custom port: 

devmemory dashboard -p 8080

Running Continuously on System Boot

Configure the DevMemory Desktop Application to launch on system boot in the application settings, or register the dashboard CLI daemon as a persistent service: 

# Register and enable the background boot daemon
devmemory enable-dashboard

# Stop and remove the background boot daemon
devmemory disable-dashboard

How it works on each OS:

  • macOS: Uses launchd — creates a Launch Agent at ~/Library/LaunchAgents/com.devmemory.dashboard.plist with KeepAlive enabled.
  • Windows: Places a startup batch file in the Startup folder for auto-launch on user login.
  • Linux: Uses systemd — creates a user service at ~/.config/systemd/user/devmemory-dashboard.service with Restart=always.

Logs are written to ~/.devmemory/dashboard.log.

License Activation GUI

The dashboard includes a dedicated License Activation interface when unactivated. In addition, when licensed, you can view your current seat usage, CPU fingerprint bindings, and JWT token validity under the dashboard settings page.

B11. Watcher ↔ MCP Coordination

To prevent duplicate analyses and save LLM token usage:

  • The MCP server and background watcher coordinate via .devmemory/watcher_state.json.
  • Every time the AI calls an MCP tool, the server updates last_mcp_activity with a timestamp.
  • The background watcher daemon detects active IDE development and suppresses automatic build triggers for a 5-minute window.
